Major Project Presentation Guide
FHIR Data Pipeline + Clinical NLP + Vector Search

Project Title
Synthea FHIR Processing and Clinical Knowledge Base

1. Project Overview

This project takes synthetic healthcare data generated by Synthea and turns it
into useful structured datasets, a searchable clinical knowledge base, and a
semantic retrieval system.

In simple words:

- Synthea creates fake patient medical records in FHIR JSON format.
- Our Python scripts read those JSON files.
- The scripts extract useful patient, condition, lab, and medication data.
- Then the project applies Clinical NLP to the note text.
- After that, the project builds a vector search system so we can search notes
  by meaning, not only by exact words.

This project shows how raw healthcare records can be transformed into an
analytics-ready and AI-ready system.

2. Main Goal of the Project

The goal is to build a complete healthcare data pipeline:

- Raw FHIR bundles
- Structured CSV datasets
- Relational SQLite knowledge base
- Clinical NLP extraction from notes
- Vector search for semantic retrieval

3. What FHIR Means

FHIR stands for Fast Healthcare Interoperability Resources.

It is a healthcare data standard used to store medical information in a
structured way. In our project, the data comes in JSON files called FHIR
Bundles. Each bundle can contain many resources such as:

- Patient
- Condition
- Observation
- MedicationRequest
- Medication
- Narrative text
- Clinical note text

4. What Has Been Completed So Far

Task 5: Parse FHIR Bundle JSON Files

We created a script called extract_data.py.

What it does:

- Reads all FHIR bundle JSON files from synthea/output/fhir/
- Extracts Patient data
- Extracts Condition data
- Extracts Observation data
- Extracts MedicationRequest data
- Resolves medication references from Medication resources
- Saves clean CSV files into datasets/

Generated files:

- patients.csv
- conditions.csv
- labs.csv
- medications.csv

Task 6: Maintain Relational Integrity

We improved the extraction process so all child records correctly point to a
valid patient.

What it does:

- Ensures patient_id exists in the patients table
- Removes broken or orphan rows
- Deduplicates patients if needed

Why this matters:

If a condition or medication does not belong to a valid patient, then the data
becomes unreliable. This task made the dataset relationally correct.

Task 7: Data Cleaning and Normalization

We standardized the extracted data.

What it does:

- Converts birthDate into age
- Converts timestamps into datetime format
- Normalizes coding systems such as SNOMED, LOINC, and RxNorm
- Separates lab observations into labs.csv

Why this matters:

Clean and standardized data is easier to analyze, query, and load into a
database.

Task 8: Create Structured Database

We created build_database.py.

What it does:

- Loads the CSV files into a SQLite database
- Creates proper database tables
- Adds primary keys and foreign keys
- Adds indexes for faster querying

Generated file:

- datasets/fhir_knowledge_base.db

Database tables:

- patients
- conditions
- labs
- medications
- clinical_notes
- clinical_entities

Why this matters:

Instead of reading many CSV files separately, we can query all the cleaned data
from one database.

Task 9: Clinical NLP Pipeline

We created extract_clinical_entities.py.

What it does:

- Reads note text from FHIR resources
- Reads narrative text from FHIR text.div
- Uses medspaCy for clinical NLP
- Uses the ConText algorithm to identify meaning such as:
  negated
  uncertain
  historical
  family history
- Saves extracted notes and entities into CSV files

Generated files:

- clinical_notes.csv
- clinical_entities.csv

Example:

Input note:
Patient denies hypertension but has chest pain.

Possible NLP understanding:

- hypertension -> negated
- chest pain -> present

Why this matters:

Clinical text is usually unstructured. NLP helps convert that text into
structured medical knowledge.

Task 10: Vector Search Knowledge Base

We created vector_search.py.

What it does:

- Converts note text into embeddings using sentence-transformers
- Stores vectors using FAISS
- Supports semantic search

Generated files:

- clinical_note_embeddings.npy
- clinical_note_index.faiss
- note_vector_metadata.csv
- vector_index_config.json

Example semantic query:

patients with chest pain but no hypertension

Why this matters:

Normal keyword search only finds exact words. Vector search finds notes with
similar meaning, even if the wording is different.

5. Final Connected Workflow

This is the full system flow:

Member 1
Synthetic Data Generation
        down
Member 2
FHIR to Structured Dataset
        down
Member 3
Clinical NLP plus Vector Search

End-to-end flow:

- Synthea generates patient bundles
- extract_data.py creates structured CSV datasets
- build_database.py builds the SQLite knowledge base
- extract_clinical_entities.py extracts note text and medical entities
- vector_search.py builds semantic retrieval over notes

Even though this is the final connected workflow, all members can work in
parallel by mocking or simulating inputs.

10. How to Run the Project

Open PowerShell in the project folder:

cd "D:\MAJOR PROJECT\Major-Project-5day"

Create virtual environment:

C:\Python312\python.exe -m venv .venv

Activate environment:

.\.venv\Scripts\Activate.ps1

Upgrade pip:

.\.venv\Scripts\python.exe -m pip install --upgrade pip

.\.venv\Scripts\python.exe -m pip install pandas medspacy sentence-transformers faiss-cpu

14. If No Real FHIR Files Are Available

If the folder synthea/output/fhir/ is empty:

- extract_data.py will say no .json files found
- extract_clinical_entities.py will also say no .json files found
- build_database.py will fail because CSV files do not exist yet
- vector_search.py may create an empty clinical_notes.csv and report no notes
  available to embed

This is normal behavior, not an error in the code.

15. Common Questions Teachers May Ask

Question:
Why did you use synthetic data instead of real hospital data?

Answer:
Real hospital data is private and sensitive. Synthea provides realistic
synthetic data that is safe for learning and experimentation.

Question:
Why use SQLite?

Answer:
SQLite is simple, lightweight, and good for a project demo. It allows us to
build a real knowledge base without requiring a separate database server.

Question:
Why use NLP in this project?

Answer:
Many important medical details are written in notes, not only in structured
fields. NLP helps us convert unstructured text into structured knowledge.

Question:
Why use vector search?

Answer:
Vector search makes it possible to search clinical notes by meaning, not just
exact keyword matches. This is useful for intelligent retrieval.

Question:
What is the role of ConText?

Answer:
ConText helps determine the status of a medical concept, such as whether it is
negated, historical, uncertain, or related to family history.
